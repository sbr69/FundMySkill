# Complete Backend Requirement Guide & Firestore Schema for "FundMySkill"

You are tasked with generating the MVP backend for this project. The frontend application is a sophisticated React application featuring a minimalist Nord-themed UI. It serves as an EdTech platform with AI Tutor functionality, adaptive video players, dynamic syllabus roadmaps, an interactive assessment/quiz engine, and a Web3 automated donation layer (Algorand Testnet USDC). 

The frontend has already been built. **Your job is to scaffold the complete backend using Firebase/Firestore and Node.js (or Next.js API Routes depending on framework choice).**

> **CRITICAL RULE ON AUTHENTICATION:** 
> Authentication must be implemented strictly in "Pass-Through Mode". The platform's login page is currently a bypass. Your endpoints must either accept an anonymous `uid` generated by the frontend client (e.g., `mock-user-xyz`) or automatically issue one. Do NOT enforce Firebase Auth barriers (email/password verification or OAuth strictness) that would block immediate frontend interaction during this MVP phase.

## 1. Firestore Database Schema definition

You must set up the exact following NoSQL collections to natively serve the frontend data hooks.

### Collection: `users`
Represents the student/creator profiles.
- **Document ID:** `{userId}`
- `name` (string)
- `avatarUrl` (string) - Profile icon URL
- `joinedAt` (timestamp)
- `bio` (string)
- `skills` (array of strings) - e.g., `["React", "Machine Learning", "Solidity"]`
- `role` (string) - `'student'` | `'creator'`

### Collection: `courses`
Holds global catalog metadata accessed by `/insider` and `/catalogue`.
- **Document ID:** `{courseId}`
- `title` (string)
- `description` (text)
- `creatorId` (string, reference to `users`)
- `thumbnailUrl` (string)
- `category` (string) - Options: `'Engineering'`, `'Computer Science'`, `'Business'`, etc.
- `level` (string) - Options: `'Beginner'`, `'Intermediate'`, `'Advanced'`
- `price` (number) - Numeric, 0 for free
- `status` (string) - `'draft'` | `'published'`
- `durationString` (string) - Frontend display e.g., `"8 Weeks"`
- `tags` (array of strings)
- `whatYouWillLearn` (array of strings)
- `requirements` (array of strings)
- **Stats map:**
  - `rating` (float)
  - `reviewsCount` (integer)
  - `enrollmentCount` (integer)

### Subcollection: `courses/{courseId}/modules`
The core roadmap payload powering the [CourseInsiderPage](file:///home/mg/binary/binary_v2/new_frontend/src/pages/CourseInsiderPage.tsx#5-473).
- **Document ID:** `{moduleId}`
- `orderIndex` (integer) - Sets chronological layout.
- `title` (string) 
- `description` (string)
- `totalDuration` (string) - e.g., `"2 hrs"`

### Subcollection: `courses/{courseId}/modules/{moduleId}/lectures`
Powering the specific [LearningInterfacePage](file:///home/mg/binary/binary_v2/new_frontend/src/pages/LearningInterfacePage.tsx#5-289).
- **Document ID:** `{lectureId}`
- `orderIndex` (integer)
- `title` (string)
- `videoUrl` (string)
- `description` (text)
- `duration` (string) - e.g., `"45 MIN"`
- **Subcollection or Array of `resources`:**
  - `title` (string) - e.g., `"Reading_Material_4.1.pdf"`
  - `type` (string) - e.g., `"PDF"`
  - `size` (string) - e.g., `"2.4 MB"`
  - `fileUrl` (string)

### Collection: `user_progress`
The most critical tracking schema. Determines what modules are `locked`, `in-progress`, or `completed` on the frontend.
- **Document ID:** `{userId}_{courseId}`
- `userId` (string, reference)
- `courseId` (string, reference)
- `enrollmentDate` (timestamp)
- `progressPercentage` (number 0-100)
- `lastAccessedModuleId` (string)
- `lastAccessedLectureId` (string)
- `completedLectures` (map) - e.g., `{ "lectureId_1": timestamp, "lectureId_2": timestamp }`
> **Backend Logic Requirement:** The endpoint serving this data must dynamically mutate module statuses. If a module has 3 lectures and the user has completed 3/3, its `status` is `'completed'`. The directly chronologically subsequent module automatically becomes `'in-progress'`, while all other chronologically later modules remain `'locked'`.

### Collection: `quizzes`
Serves the [CourseQuizPage](file:///home/mg/binary/binary_v2/new_frontend/src/pages/CourseQuizPage.tsx#12-220).
- **Document ID:** `{quizId}`
- `contextSource` (map) - `{ type: 'module'|'lecture', id: string }`
- `title` (string) - e.g., `"Weekly Synthesis Quiz"`
- `timeLimitSeconds` (integer) - e.g., `900`
- `questions` (array of maps):
  - [id](file:///home/mg/binary/binary_v2/new_frontend/src/components/layout/SideNavBar.tsx#10-102) (string)
  - `text` (string)
  - `points` (integer)
  - `options` (array of maps):
    - `letter` (char)
    - `text` (string)
    - `isCorrect` (boolean)
  - `immediateFeedback` (string) - Appears dynamically on click in the UI.

### Collection: `quiz_attempts`
- **Document ID:** Auto-generated
- `quizId` (string)
- `userId` (string)
- `score` (integer)
- `accuracyPercentage` (number)
- `timeTakenSeconds` (integer)
- `answers` (map) - e.g., `{"questionId_1": "A"}`
- `createdAt` (timestamp)

### Collection: `ai_sessions`
Context-aware window tracking the [LearningInterfacePage](file:///home/mg/binary/binary_v2/new_frontend/src/pages/LearningInterfacePage.tsx#5-289) sliding chat panel.
- **Document ID:** Auto-generated
- `userId` (string)
- `lectureId` (string) - Crucial for limiting RAG scope.
- `messages` (array or subcollection):
  - `role` (`'user'` | `'ai'`)
  - `content` (string)
  - `timestamp` (timestamp)

### Collection: `donations`
A transparent verification ledger capturing the Algorand Web3 action in [DonationPage.tsx](file:///home/mg/binary/binary_v2/new_frontend/src/pages/DonationPage.tsx).
- **Document ID:** `{txHash}` (The unique Algorand transaction hash)
- `donorAddress` (string) - Sender's Algorand wallet address.
- `recipientAddress` (string) - Course Creator / Platform address.
- `amountUSDC` (number)
- `message` (string)
- `donorUserId` (string, optional reference to `users`)
- `verifiedAt` (timestamp)
- `status` (`'pending'`, `'confirmed'`, `'failed'`)

## 2. API Endpoints / RPC Functions Required

### A. Core Discovery & Access
1. **`GET /api/courses`**
   - **Query Params:** `category`, `level`, `priceMax`, `search`
   - **Returns:** Paginated lists of `courses` schema to populate [CourseCataloguePage](file:///home/mg/binary/binary_v2/new_frontend/src/pages/CourseCataloguePage.tsx#158-363).
2. **`GET /api/courses/:id`**
   - **Returns:** Full snapshot for [CourseOverviewPage](file:///home/mg/binary/binary_v2/new_frontend/src/pages/CourseOverviewPage.tsx#5-310). Includes nested module count aggregation.
3. **`POST /api/courses/:id/enroll`**
   - **Body:** `{ userId }`
   - **Action:** Generates a new `user_progress` document initializing the user at Module 1, Lecture 1.

### B. Syllabus & Progress Tracking
1. **`GET /api/progress/:courseId`**
   - **Query Params:** `userId`
   - **Action:** Fetches the entire module structure, calculating on-the-fly (`completed` | `in-progress` | `locked`) flags per module based on the user's `completedLectures` array.
2. **`POST /api/progress/lecture/complete`**
   - **Body:** `{ userId, courseId, lectureId }`
   - **Action:** Appends the `lectureId` to the `completedLectures` map. Triggers recalculation of the `progressPercentage`.

### C. AI Integrations (The USP of the App)
1. **`POST /api/ai/tutor/chat`**
   - **Body:** `{ userId, lectureId, message }`
   - **Action:** Fetches the `lectureId`'s transcript and `resources` texts. Pipes system prompt + history + RAG context into an LLM (e.g. Gemini 1.5/GPT-4o).
   - **Returns:** The AI's response text (streamed if possible) along with 2 dynamically generated `{suggestedPrompts}` based on the response.
2. **`POST /api/ai/quiz/generate`**
   - **Body:** `{ userId, lectureId }`
   - **Action:** Wired to the precise "Generate AI Quiz" button in the frontend. Scrapes the lecture context, sends a forced JSON schema prompt to an LLM: *"Generate a 5-question multiple choice quiz on [lecture contextual data] with deeply informative immediate negative/positive feedback."*
   - **Returns:** The newly spawned `quizId` which the frontend uses to `navigate('/quiz/ai-generated-{quizId}')`.

### D. Web3 Ledger Interaction
1. **`POST /api/donations/verify`**
   - **Body:** `{ txHash, sender, amount, customMessage, userId }`
   - **Action:** Hits the Algorand testnet API (algoexplorer/purestake equivalent standard Node API) to blindly verify that `txHash` exists, that it originated from `sender`, sent `amount` ASAs (USDC) to the platform's central treasury address.
   - **Success:** Commits to `donations` collection.
2. **`GET /api/donations/leaderboard`**
   - **Returns:** Top N un-anonymous rows indexing highest `amountUSDC` to show in Community Feeds.

## 3. Implementation Steps for the Backend Agent
1. **Initialize Firebase Admin SDK** inside a standard Express project or a Next.js `/api` routing construct.
2. **Write a Mock Data Seeder script** immediately. Populate Firestore with 2 dummy users, 3 comprehensive courses (using the nested modules/lectures schema). *This is vital as the frontend requires immediate complex hierarchical JSON.*
3. Ensure CORS policies allow localhost frontend dev variants. 
4. **Mock the Auth middleware** cleanly (e.g. `req.user = req.headers['x-mock-user-id'] || 'mock-fallback-user'`).
5. Structure your handlers to be as dumb and fast as possible, returning exactly what the frontend anticipates.
